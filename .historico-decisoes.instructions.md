# Instruções de Formato: Histórico de Decisões

## Estrutura Geral

O arquivo `historico-decisoes.md` documenta decisões arquiteturais e de design agrupadas por escopo.

### Hierarquia de Heading

```
## Histórico de Decisões             ← Título principal (H2)

# ARQUITETURA & HIERARQUIA          ← Escopo/seção (H1)

## 1. Titulo Descritivo ✓            ← Decisão individual (H2 com número ordinal e ícone)
```

**Regra importante:** NÃO use `---` (linha horizontal) para separar decisões ou seções. Use apenas quebras de linha entre blocos.

---

## Formato de cada Decisão

Cada decisão segue esta estrutura:

```markdown
## [N]. [Título Descritivo] [ícone-de-status]

**Decisão:** [1-2 frases diretas e afirmativas descrevendo o que foi decidido]

**Contexto:** 
- [Bullet point com informação contextual]
- [Razão pela qual a decisão foi necessária]
- [Observação sobre alternativas consideradas]

**Justificativa:** [Opcional] [Raciocínio da escolha — omitir se autoexplicativo]

**Consequências:** [Opcional] [Impactos, riscos, trade-offs — omitir se óbvio]
```

### Componentes Obrigatórios

| Campo | Obrigatório? | Descrição |
|-------|--------------|-----------|
| **Número ordinal** | ✓ | Começa em 1, incrementa sequencialmente |
| **Título** | ✓ | Descritivo, máx 60 caracteres |
| **Ícone de status** | ✓ | `✓` (ativo) ou `⏳` (pendente) |
| **Decisão** | ✓ | Afirmação clara do que foi decidido |
| **Contexto** | ✓ | Sempre presente; pode ser parágrafo ou bullets |
| **Justificativa** | ✗ | Omitir se autoexplicativo ou redundante com Contexto |
| **Consequências** | ✗ | Omitir se óbvio; destacar trade-offs e riscos |

### Componentes Opcionais

**Justificativa** é omitida quando:
- O raciocínio já está claro no Contexto
- Não há informação adicional significativa
- Decisão é autoexplicativa

**Consequências** é omitida quando:
- Os impactos são óbvios ou triviais
- Não há trade-offs relevantes
- Não há efeitos colaterais significativos

## Escopos (H1)

Agrupe decisões relacionadas sob escopos temáticos em H1:

- `# ARQUITETURA & HIERARQUIA` — Estrutura de pastas, árvores, IDs
- `# CAMPOS & MODELOS` — Tipos de campos, modelos, observação
- `# BUSCA & INDEXAÇÃO` — Busca, importação, ordenação
- `# SEGURANÇA & CRIPTOGRAFIA` — Criptografia, senhas, proteção
- `# ARQUIVOS & CONCORRÊNCIA` — Persistência, conflitos, corrupção
- `# DESIGN & RESTRIÇÕES` — Limites, bom senso, configurações
- `# ESCOPO v1` — Fora de escopo, pendências

Adicione novos escopos conforme necessário. **Sempre coloque H1 escopos antes da primeira decisão.**

## Ícones de Status

| Ícone | Significado | Uso |
|-------|-------------|-----|
| `✓` | Vigente, aprovado, implementado | Decisão ativa |
| `⏳` | Pendente, em discussão, futuro | Decisão v2, experimental |
| `❌` | Rejeitado, descartado | Não use no histórico ativo; arquive |

## Exemplo Completo

```markdown
## 5. Identidade por UUID (modelos/segredos), por nome (pastas) ✓

**Decisão:** 
- Pastas identificadas por nome (unicidade dentro da pasta pai)
- Modelos e segredos identificados por UUID gerado pelo sistema

**Contexto:**
- Permite dois segredos com mesmo nome (em contextos diferentes)
- Simplifica import/rename sem quebrar vínculo
- Nomes são apenas metadados descritivos, não identificadores

**Justificativa:** Flexibilidade (renomear sem perder referência, importar dupes); nomes duplicados podem confundir, mas UX mitiga com contexto visual + ordenação inteligente + affordances.

**Consequências:** 
- UX deve desambiguar visualmente (contexto, cores, ícones)
- Alternativa (impor unicidade) criaria atrito desnecessário (renomear para poder importar, banir "Login"/"Login" em contextos diferentes)
```

## Regras de Formatação

### Sem separadores horizontais

❌ **NÃO FAÇA:**
```markdown
## 1. Decisão A ✓

**Decisão:** ...

---

## 2. Decisão B ✓
```

✅ **FAÇA:**
```markdown
## 1. Decisão A ✓

**Decisão:** ...


## 2. Decisão B ✓
```

Use **duas quebras de linha** entre decisões; não use `---`.

### Bullets em Contexto/Consequências

✅ Use bullets quando houver múltiplos pontos:
```markdown
**Contexto:**
- Primeiro ponto
- Segundo ponto
- Terceiro ponto
```

✅ Use parágrafo quando houver um ponto único ou narrativa:
```markdown
**Justificativa:** Segurança vem de explicitação — usuário entende claramente...
```

### Nomes de componentes

Sempre use `**palavra-chave:**` (bold seguido de dois-pontos):
- `**Decisão:**`
- `**Contexto:**`
- `**Justificativa:**`
- `**Consequências:**`

### Destaques especiais

Use negrito para termos técnicos ou importantes:
```markdown
**Decisão:** Usar **mlock/VirtualLock** para impedir swap...
```

Use itálico apenas para notas parentéticas:
```markdown
**Contexto:** Alternativas consideradas: "dado" (*genérico*), "detalhe"...
```

## Numeração e Ordem

- Números começam em **1** e incrementam sequencialmente
- **Não renumere** quando remover decisão anterior (cria instabilidade)
- Se remover decisão: archive em seção "Descartadas" ou remova sem renumeração
- Adicione novas decisões ao **final** de seu escopo

```markdown
# EXEMPLO

## 1. Primeira decisão ✓
## 2. Segunda decisão ✓
## 4. Quarta decisão ✓  ← Pulou 3 (antiga foi removida)
## 5. Quinta decisão ✓
```

## Tamanho e Concisão

| Componente | Mínimo | Máximo | Exemplo |
|-----------|--------|--------|---------|
| Título | 5 palavras | 10 palavras | "Pasta Geral como raiz não-renomeável" |
| Decisão | 1 linha | 3 linhas | Afirmação clara e direta |
| Contexto | 2 bullets | 6 bullets | Motivos, alternativas, observações |
| Justificativa | 1 frase | 3 frases | Raciocínio do porquê |
| Consequências | 2 pontos | 5 pontos | Trade-offs, riscos, impactos |

**Objetivo:** Legível para humanos E escanável para IA (balanceado).

## Checklist para Nova Decisão

Antes de adicionar uma decisão nova, valide:

- [ ] Título é descritivo e único dentro do escopo?
- [ ] Ícone de status está presente (`✓` ou `⏳`)?
- [ ] **Decisão** é uma afirmação clara (não pergunta)?
- [ ] **Contexto** explica o porquê (não assume conhecimento)?
- [ ] **Justificativa** foi omitida ou adicionada apropriadamente?
- [ ] **Consequências** destaca trade-offs ou riscos relevantes?
- [ ] Sem uso de `---` para separar desta decisão?
- [ ] Formatação segue bold `**palavra:**` para rótulos?
- [ ] Numeração é sequencial (n-1 + 1)?
- [ ] Decisão está no escopo correto (H1)?
- [ ] Sem typos ou formatação quebrada?

## Versionamento

As decisões refletem o estado atual do design. Quando uma decisão mudar:

1. **Mude o ícone** de `✓` para `⏳` se passar para discussão
2. **Atualize Contexto/Justificativa** se justificativa mudar
3. **Atualize Consequências** se impactos forem revisados
4. **Não delete** decisões antigas (histórico é valioso)
5. **Crie nova decisão** se muda fundamentalmente (não edit)

Exemplo de evolução:
```markdown
## 15. Proteção de senha em memória ⏳  ← mudou de ✓ para ⏳

**Decisão:** Reavaliando trade-off senha em memória vs. UX...

**Contexto:** [atualizado com nova informação]
```

O arquivo `historico-decisoes.md` funciona como **referência arquitetural** para `requisitos.md`:

- Se requisito contradiz decisão → atualize ambos
- Se nova decisão afeta requisitos → propague a mudança

## 1. Pasta "Geral" como raiz não-renomeável ✓

- Pasta Geral garante destino sempre disponível

**Contexto:**

**Consequências:** Modelo conceitual simples; sem ambiguidade.

## FAQ

**P: Por que não usar `---` para separar?**
R: Reduz ruído visual, melhora scaneabilidade. Quebras de linha duplas são suficientes.

**P: Quando devo usar Justificativa?**
R: Quando há raciocínio adicional não coberto pelo Contexto. Se Contexto já explica o porquê, omita.

**P: Posso reordenar decisões dentro de um escopo?**
R: Sim, mas mantenha numeração estável (não renumere). Use ordem temática dentro do escopo.

**P: Como documentar decisão rejeitada?**
R: Mude ícone para `❌` ou crie seção "Descartadas" separada. Não mantenha no histórico ativo.

**P: Preciso datar as decisões?**
R: Não. Histórico de git fornece data. Se data é relevante, adicione em Contexto.

## Compatibilidade com IA/LLM

Formato foi otimizado para:
- **Parsing**: Headings H1/H2, bold `**:**` como campos estruturados
- **Contextualização**: Decisão + Contexto + Justificativa fornecem visão 360°
- **Referência rápida**: Números + Títulos permitem busca "Decisão #5"
- **Integração**: Estrutura markdown pura (sem markup exótico)
- **Legibilidade humana**: Sem separadores excessivos, sem formatação poluída
