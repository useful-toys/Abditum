# Phase 06.3: Reimplement file picker - Context

**Gathered:** 2026-04-07 (assumptions mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the existing placeholder `filePickerModal` (currently embedded in `dialogs.go`) with a spec-compliant implementation of the two-panel file picker defined in `tui-specification-novo.md` §FilePicker.

The current implementation is a simplified stub: flat directory list (no tree), no path header, no proper borders/junctions, wrong metadata format, wrong initial focus, no MessageManager wiring, and Open/Save modes are not differentiated. This phase makes the picker fully spec-compliant.

This phase implements:
- **`filePickerModal`** extracted into `internal/tui/filepicker.go` — spec-compliant two-panel modal
- **Lazy recursive tree** (`treeNode` struct) with expand/collapse, `▶`/`▼`/`▷` indicators, 2-space indentation per level
- **Open mode:** Tree → Files (2 Tab stops); `Enter` on file confirms; cursor movement in tree updates file panel in real-time
- **Save mode:** Tree → Files → Campo Nome (3 Tab stops); `Enter` on file copies name to field (does not confirm); `bubbles/textinput` filename field
- **Spec-accurate borders:** `border.focused` main border; `├┬┴┤` junctions with `border.default`; scroll indicators `↑`/`↓`/`■` replace border chars on the respective panel side
- **Path header:** `Caminho: /path/to/dir` — updates as tree cursor moves
- **Metadata format:** `dd/mm/aa HH:MM` (not relative)
- **MessageManager wiring:** hints from the flow's `*MessageManager` passed at construction time
- **Semantic mode enum:** `FilePickerOpen | FilePickerSave` replacing `FilePickerFile/Directory/Any`
- **Updated tests** in `filepicker_test.go` to match new behavior (dates, sizes, focus, struct shape)

This phase does NOT implement:
- Mouse click/double-click interactions (spec documents them but they are lower priority; keyboard nav is the MVP)
- `special.highlight` token as a new Theme field — use hardcoded Tokyo Night hex for the selected file row background until Phase 9 (theme system)
- Any changes to password modals, flows, or other TUI components
</domain>

## Implementation Decisions

### Architectural Classification

- **D-00: `filePickerModal` is a functional modal dialog — implements `modalView`, not `childModel`**
  - The spec classifies FilePicker under §Diálogos Funcionais alongside PasswordEntry and PasswordCreate.
  - `filePickerModal` must satisfy the **`modalView` interface** as defined in `tui-elm-architecture.md`:
    ```go
    type modalView interface {
        Update(tea.Msg) tea.Cmd
        View() string        // returns string, NOT tea.View — see note below
        Shortcuts() []Shortcut
        SetSize(w, h int)
    }
    ```
  - **`View() string` is correct for `modalView`** — `arquitetura.md §1` warns that `tea.Model.View()` returns `tea.View` struct (a Bubble Tea v2 change). That warning applies exclusively to `rootModel`, which implements `tea.Model` and calls `tea.NewView(content)`. The `modalView` interface is an internal project contract, completely separate from `tea.Model`. `filePickerModal.View()` returns `string` (the rendered box) — `rootModel` assembles it into the frame and wraps it in `tea.View`. Confirmed in `root.go`: `v := tea.NewView(content)` is called in `rootModel.View()`, not in `modalView.View()`.
  - **`Init()` is NOT part of `modalView`** and is NOT called by `rootModel` on push. The `rootModel` handler for `pushModalMsg` only appends the modal to the stack — confirmed in `root.go` lines 185–190.
  - `filePickerModal` has its own `Init() tea.Cmd` method (not the interface). It is called by the factory/flow before the push — see D-14 for the full lifecycle.
  - It is pushed onto the modal stack via `pushModalMsg{modal: fpk}` and dismissed via `popModalMsg{}` — the same lifecycle as all other modals.
  - `rootModel.View()` calls `top.SetSize(m.width, workH)` on the top modal before calling `top.View()` — the modal can rely on having valid dimensions by the time `View()` is first invoked.
  - It does **not** implement `childModel` (no `Render(width, height int) string` variant; it is not a persistent work area).
  - **No border.default:** functional dialogs use `border.focused` (see D-09). Decision dialogs use `border.default` with semantic severity colors — FilePicker is explicitly not a decision dialog.

### Tree Data Structure

- **D-01: Lazy recursive `treeNode` struct for the Estrutura panel**
  - `treeNode` struct: `path string`, `name string`, `depth int`, `expanded bool`, `loaded bool`, `children []*treeNode`, `hasSubdirs bool` (for `▷` detection).
  - Lazy loading: `os.ReadDir` is called on a node only when it is first expanded (`expanded = true && !loaded`). After loading, `loaded = true`.
  - **Visible items list:** a `[]visibleNode` (flattened view of expanded tree) is rebuilt after every expand/collapse and on init. Navigation (`↑↓`) operates on this flat slice.
  - **Initial tree state:** expand only the path from `/` down to the initial directory (CWD or fallback). All other nodes start collapsed.
  - **▷ indicator:** a directory shows `▷` when it has no subdirectories (but may still contain `.abditum` files). Detected at load time by checking if `os.ReadDir` returns any `IsDir()` entries.
  - **Cursor movement auto-updates file panel:** moving the tree cursor (even without expanding) updates `currentPath` to the node under the cursor and reloads the file list.

### Mode Enum

- **D-02: Replace `FilePickerMode` with semantic `FilePickerOpen | FilePickerSave`**
  - Remove constants `FilePickerFile`, `FilePickerDirectory`, `FilePickerAny`.
  - Add `FilePickerOpen FilePickerMode = iota` and `FilePickerSave`.
  - Update 3 call sites:
    - `flow_open_vault.go` — push `&filePickerModal{mode: FilePickerOpen}`
    - `flow_create_vault.go` — push `&filePickerModal{mode: FilePickerSave}`
    - `flow_save_and_exit.go` — push `&filePickerModal{mode: FilePickerSave}`
  - The `FilePicker(title, mode, ext)` factory in `dialogs.go` can be updated or removed if callers push directly.

### MessageManager Wiring

- **D-03: Receive `*MessageManager` from the flow at construction time**
  - `filePickerModal` struct gains a `messages *MessageManager` field.
  - Callers pass the flow's `messages` when constructing the modal:
    ```go
    &filePickerModal{mode: FilePickerOpen, messages: f.messages}
    ```
  - Hints are emitted via `m.messages.Show(MsgHint, text, 0, true)` on every focus change (panel switch, file selection change).
  - Messages per spec:
    - Open — tree focus: `• Navegue pelas pastas e selecione um cofre`
    - Open — files focus (with selection): `• Selecione o cofre para abrir`
    - Open — files focus (empty panel): `• Nenhum cofre neste diretório — navegue para outra pasta`
    - Save — tree focus: `• Navegue pelas pastas e escolha onde salvar`
    - Save — files focus: `• Arquivos existentes neste diretório`
    - Save — field focus (empty): `• Digite o nome do arquivo — {m.ext} será adicionado automaticamente` (e.g. `.abditum será adicionado automaticamente` — spec uses `<ext>` placeholder, render with actual `m.ext` value)
    - Save — field focus (filled): `• Confirme para salvar o cofre`

### File Location

- **D-04: Extract `filePickerModal` into `internal/tui/filepicker.go`**
  - Remove the `filePickerModal` struct and all its methods from `dialogs.go`.

  - `filepicker_test.go` already exists and imports `package tui` — no test file relocation needed.

### Metadata Format

- **D-05: Absolute date/time format `dd/mm/aa HH:MM` (spec format)**
  - Go format string: `"02/01/06 15:04"` applied to `fileInfo.ModTime().Local()`.
  - Replaces the current `formatRelativeDate()` function (remove it).
  - Retains `formatFileSize()` but updates to space-separated format: `25.8 MB`, `1.2 KB` (not `25.8MB`).

### Tab Stop Order and Confirmation

- **D-06: Tab cycles differ by mode; Enter semantics differ on file panel**
  - Open mode: Tab cycles Tree (0) → Files (1) → Tree (0). 2 stops.
  - Save mode: Tab cycles Tree (0) → Files (1) → Campo Nome (2) → Tree (0). 3 stops.
  - `Enter` on file panel — Open mode: confirms selection (emits `filePickerResult{Path}`), pops modal.
  - `Enter` on file panel — Save mode: copies filename (without extension) into the textinput field, moves focus to campo nome (focusPanel = 2). Does NOT confirm.
  - `Enter` on campo nome (Save mode): confirms if field non-empty (emits `filePickerResult{Path}` with `.abditum` appended if missing).
  - `Enter` on tree panel: if current node contains `.abditum` files, move focus to Files panel. Otherwise no-op.
  - Initial focus: Tree (focusPanel = 0) for both modes.

### Layout and Sizing

- **D-08: Modal dimensions, proportions, and structural borders**
  - **Width:** `min(70, 80% of terminal width)` columns.
  - **Height:** `80% of terminal height` rows.
  - **Panel split:** tree panel ~40% of inner width; files panel ~60%.
  - **Vertical padding:** 0 (spec exception — justified by density; `╭╮╰╯` + internal headers provide containment).
  - **Horizontal padding:** 2 columns inside the main border.
  - **Title:** `Abrir cofre` (Open mode) / `Salvar cofre` (Save mode).
  - **Bottom border action layout:**
    - Open: `╰── Enter Abrir ───────────────┴────────────────────── Esc Cancelar ──╯`
    - Save: `╰── Enter Salvar ───────────────────────────────────────── Esc Cancelar ──╯`
    - `┴` marks the junction of the panel separator with the base — only present in Open mode (no filename panel below).
    - In Save mode the bottom border has no `┴` (the `├─┴─┤` separator above the filename field provides the junction).
  - **Action `Enter Abrir`/`Enter Salvar` state:**
    - Open — blocked (`text.disabled`) when the directory under the tree cursor contains no `.abditum` files; active (`accent.primary` bold) when it does — **even if focus is on the Tree panel**.
    - Save — blocked when filename field is empty; active when non-empty.
  - **Scroll indicators replace border chars:**
    - Tree panel scroll: `↑`/`■`/`↓` replace the `│` of the panel separator (right edge of tree).
    - Files panel scroll: `↑`/`■`/`↓` replace the `│` of the modal's right border.
  - **Command bar while FilePicker is active:** only `Tab Painel` and `F1 Ajuda` — all other actions hidden.

### Sort, Metadata Alignment, and Truncation

- **D-11: Ordering, column layout, and truncation priority**
  - **Sort order — tree:** alphabetical, case-insensitive (lexicographic, `a` == `A`).
  - **Sort order — files:** alphabetical, case-insensitive by name without `.abditum` extension.
  - **Tree indentation:** 2 spaces per depth level.
  - **Metadata column layout in files panel:**
    - Name occupies the left portion; size is right-aligned; date/time is at a fixed position to the right of size.
    - Format: `● {name:<N}  {size:>8}  {datetime}` — exact column widths are implementation detail; intent is vertical scanning alignment.
  - **Truncation priority in narrow terminals:** metadata (date first, then size) truncated before filename. Filename only truncates if no space remains even for it.
  - **Path header truncation:** `Caminho:` value truncated from the **left** with `…` (e.g., `…/projetos/abditum`) when it exceeds available width.
  - **Size format:** base-1024, 1 decimal place, space before unit — `25.8 MB`, `1.2 KB`, `512 B`. Units: B, KB, MB, GB.

### Save Mode Field Behavior

- **D-12: Filename field semantics in Save mode**
  - **`↑↓` in files panel** only moves the highlight — does **not** copy the filename to the field. Only `Enter` (on a file row) copies the name (without `m.ext` extension) and moves focus to Campo Nome.
  - **Field preserved on tree navigation:** navigating the tree (moving cursor, expanding/collapsing) does **not** clear the filename field. Whatever the user typed stays.
  - **Invalid chars:** `/ \ : * ? " < > |` are silently blocked in the textinput — no error message.
  - **Extension appended:** `m.ext` is added silently to the returned `filePickerResult.Path` if the typed name does not already end in `m.ext`. The field text itself is **not** modified — the extension appears only in the emitted result.
  - **`Nome sugerido` parameter:** if the flow passes a suggested name at construction, it pre-fills the textinput field (without extension in the display). Currently no flow passes a suggested name — field starts empty.

### Directory Permission Errors

- **D-22: Directories without read permission are shown in the tree but cannot be expanded; error emitted via MessageManager**
  - Directories without read permission are **shown normally** in the tree (visible, cursor can land on them).
  - When the user tries to expand such a directory (`→` key or expand action): `os.ReadDir` returns an error → the directory **remains collapsed** (`expanded = false`); no children are added.
  - The error is surfaced via `m.messages.Show(MsgError, "✕ Sem permissão para acessar "+dirName, 5*time.Second, true)`. The `dirName` used in the message is the **directory name only** (not the full path) to avoid leaking path information per `arquitetura.md §5` (logs e privacidade).
  - The file panel for an inaccessible directory shows no files (empty state, same as a directory with no `<ext>` files).
  - Transition table entry: `Tentar expandir pasta sem permissão → Erro na barra (✕ Sem permissão para acessar <pasta>); pasta permanece recolhida`.

- **D-21: Golden tests use 60×24 (standard) and 30×24 (narrow), following `arquitetura-teste.md` — not the 80×24 mentioned in `arquitetura.md §8`**

  - Therefore: all 8 golden file pairs in D-07 use `SetSize(60, 24)` or `SetSize(30, 24)` via fixture constructors — consistent with `decision_test.go` pattern.

### File Filtering and Display Rules

- **D-13: Extension filter is a required field `ext string`, passed at construction; hidden files excluded; extension omitted in display**
  - The spec defines `extensao` as a required input parameter (String, Obrigatório: Sim). `filePickerModal` must have an `ext string` field set at construction — it is **not** hardcoded.
  - Only files ending in `m.ext` are shown in the files panel. No other file types are shown. There is no configurable filter field.
  - All 3 current call sites pass `.abditum` as the extension — this is always the value in practice, but the field must exist and be used for correctness per the spec contract.
  - Files and directories whose name starts with `.` (hidden) are **never shown** — in either panel.
  - `m.ext` is **omitted from display** in the files panel (redundant with the filter). The `filePickerModal.files` slice stores names without the extension; the full path reconstructed for `filePickerResult.Path` re-adds `m.ext` if not already present.
  - The `ext` parameter in the `FilePicker()` factory is **kept** (not removed) and passed through to `filePickerModal.ext` at construction.

### Init and CWD Fallback

- **D-14: `Init()` is called by the factory inside its `tea.Cmd` closure, before `pushModalMsg`**
  - `filePickerModal` has an `Init() tea.Cmd` method defined on itself (NOT part of the `modalView` interface). It follows the same pattern as `PasswordEntry`/`PasswordCreate` factories in `dialogs.go`: the factory constructs the modal, calls `Init()`, and **batches** `Init()`'s returned cmd alongside `pushModalMsg`:
    ```go
    // inside the factory tea.Cmd closure:
    fpk := &filePickerModal{mode: mode, ext: ext, messages: messages, theme: theme}
    cmd := fpk.Init()  // must NOT discard return value (unlike the current broken FilePicker() factory)
    return tea.Batch(
        func() tea.Msg { return pushModalMsg{modal: fpk} },
        cmd,
    )
    ```
  - Since `rootModel` does NOT call `Init()` on push, all initialization must complete synchronously inside this closure.
  - `Init()` performs this sequence:
    1. Set `focusPanel = 0` (tree) for both modes.
    2. If Save mode and `suggestedName` is non-empty, pre-fill the textinput field with it (without `.abditum` extension).
    3. Resolve `currentPath` via this fallback chain:
       - Try `os.Getwd()`. If readable (`os.ReadDir` succeeds) → use it.
       - Else fall back to `os.UserHomeDir()`. Return a `tea.Cmd` that emits `MsgWarn`: `⚠ Diretório atual inacessível — navegando para home`.
    4. Expand the path from `/` down to `currentPath` building the initial `treeNode` chain.
    5. Call `loadFilesForCursor()` to populate the initial file list and set `fileCursor`.
  - `theme` field is set at construction (`&filePickerModal{..., theme: theme}`); if nil at render time, fall back to `ThemeTokyoNight`.

### Auto-Pre-Selection in Files Panel

- **D-15: First `.abditum` file auto-selected when tree cursor moves to a dir that has files**
  - Whenever `currentPath` changes (cursor move, expand, or collapse), the files panel reloads.
  - If the new `currentPath` has at least one `.abditum` file: `fileCursor` is set to 0 (first file, alphabetically). The action default (`Enter Abrir`) becomes active.
  - If the new `currentPath` has no `.abditum` files: `fileCursor` is set to -1 (no selection). The files panel shows `Nenhum cofre neste diretório`. The action default becomes blocked (`text.disabled`).
  - This auto-selection applies even when focus is on the Tree panel — the action default state always reflects what is in the files panel for the current cursor dir.

### Keyboard Gap: `←` on Already-Collapsed Node

- **D-16: `←` on a node that is already collapsed (and not root) navigates to parent**
  - Spec does not explicitly define this case; the standard file picker convention is applied: pressing `←` on an already-collapsed node moves the cursor to its parent node (if any). This is consistent with how macOS Finder and VS Code tree views behave.
  - At root `/`: `←` is always no-op regardless of expansion state.

### Theme Field Requirement

- **D-17: `filePickerModal` requires a `theme *Theme` field; passed at construction**
  - `filePickerModal` struct must have a `theme *Theme` field.
  - Callers must pass `theme` at construction: `&filePickerModal{mode: FilePickerOpen, messages: f.messages, theme: f.theme}`.
  - If `theme` is nil at render time, fall back to `ThemeTokyoNight` (same defensive pattern as other modals).
  - The `FilePicker()` factory in `dialogs.go` is updated to accept `theme *Theme` and `messages *MessageManager` as parameters.

### `Shortcuts()` Return Value

- **D-18: `Shortcuts()` returns exactly `Tab Painel` and `F1 Ajuda`**
  - While `filePickerModal` is the active modal, `rootModel` calls `Shortcuts()` to populate the command bar.
  - `Shortcuts()` must return:
    ```go
    []Shortcut{
        {Key: "Tab", Label: "Painel"},
        {Key: "F1",  Label: "Ajuda"},
    }
    ```
  - `Enter`/`Esc` are **not** included in `Shortcuts()` — they are rendered in the modal's own border, not the command bar (per spec).

### `filePickerResult` When Cancelled

- **D-19: `filePickerResult.Path` is empty string when `Cancelled: true`**
  - When the user presses `Esc`, the modal emits `filePickerResult{Cancelled: true, Path: ""}`.
  - Callers (`openVaultFlow`, `createVaultFlow`, `saveAndExitFlow`) check `msg.Cancelled` first and call `endFlow()` immediately — they never read `Path` when `Cancelled` is true.
  - No change to the `filePickerResult` struct definition.

### `Caminho:` Header Always Visible

- **D-20: Path header line `Caminho: {path}` is always visible, always read-only**
  - The `Caminho:` header occupies the first content row below the top border, above the `├─ Estrutura ──┬─ Arquivos ──┤` separator.
  - It is always rendered regardless of mode (Open or Save) and regardless of panel focus.
  - Value displayed is the path of the **directory under the tree cursor** (not the selected file's parent, not the expanded node — the cursor node itself).
  - The label `Caminho:` uses `text.secondary`; the path value uses `text.primary` (per D-09).
</decisions>
