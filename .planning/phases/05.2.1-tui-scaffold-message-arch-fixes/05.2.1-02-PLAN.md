  <action>
    Add truncation logic to RenderCommandBar() in internal/tui/actions.go.

    Current behavior (lines 172-222): Renders ALL body actions + F1 anchor, no width checking. If body + anchor exceeds terminal width, the bar overflows.

    New behavior (D-11 through D-16):

    1. F1 anchor always stays visible — reserve space for "F1 Ajuda" before distributing body actions (D-12)

    2. Calculate available width for body:
       - anchorWidth = lipgloss.Width of "F1 Ajuda" (keyStyle + " " + labelStyle)
       - availableForBody = width - anchorWidth - minimum gap (at least 1 space between body and anchor)
       - If availableForBody <= 0 → show only F1 anchor (D-14)

    3. Truncate body actions by priority (D-13):
       - bodyActions are already sorted by Priority descending from Visible()
       - Build body parts one at a time, tracking cumulative width
       - Each part = keyStyle.Render(key) + " " + labelStyle.Render(label)
       - Separators " · " (sepStyle.Render(" · ")) between parts — include separator width in fit calculation (D-15)
       - First part has no separator before it
       - Stop adding parts when next part + separator would exceed availableForBody

    4. Body padding: "  " (2 spaces) at start of body string (existing behavior)

    5. Recalculate on every render (D-16): No caching — width parameter changes on terminal resize, Visible() may change as actions are enabled/disabled

    Implementation approach — replace lines 191-221 with:

    ```go
    // Build anchor string first to reserve space
    anchor := keyStyle.Render(anchorAction.Keys[0]) + " " + labelStyle.Render(anchorAction.Label)
    anchorW := lipgloss.Width(anchor)

    // Build body parts with truncation
    var bodyParts []string
    bodyUsed := 0
    sep := sepStyle.Render(" · ")
    sepW := lipgloss.Width(sep)

    for _, act := range bodyActions {
        if len(act.Keys) == 0 {
            continue
        }
        part := keyStyle.Render(act.Keys[0]) + " " + labelStyle.Render(act.Label)
        partW := lipgloss.Width(part)

        // Calculate width if we add this part
        needed := partW
        if len(bodyParts) > 0 {
            needed += sepW // separator before this part
        }

        if bodyUsed+needed > width-anchorW-1 { // -1 for minimum gap between body and anchor
            break // drop this and all remaining (lower priority) actions
        }

        bodyParts = append(bodyParts, part)
        bodyUsed += needed
    }

    var body string
    if len(bodyParts) > 0 {
        body = "  " + strings.Join(bodyParts, sep)
    }

    // Position anchor at right edge
    bodyW := lipgloss.Width(body)
    gap := width - bodyW - anchorW
    if gap < 1 {
        gap = 1
    }
    return body + strings.Repeat(" ", gap) + anchor
    ```

    Note: The existing code at lines 180-189 already separates F1 anchor from bodyActions — keep that logic. Only replace the body building and positioning logic (lines 191-221).

    **Changes to pocMessagesModel struct:**
    - Remove `showHelp bool` field
    - Add `modals []tui.modalView` field — but modalView is unexported, so instead add a field that can hold the help modal
    - Since modalView is unexported and poc_model.go is in package `main` (not `tui`), we cannot directly store modalView values
    - Alternative approach: Use the tui package's helpModal through its public interface. The helpModal satisfies modalView which has Update(), View(), Shortcuts() — but these are unexported interface methods
    - Practical approach for PoC: Store the help modal as `any` and type-assert to a local interface matching the modalView contract, OR simply call the help modal's methods through a wrapper

    Actually — the simplest approach that aligns with rootModel architecture (D-19):
    - Add `modals []any` field to pocMessagesModel (stores modalView instances as any)
    - F1 handler dispatches: `m.modals = append(m.modals, tui.NewHelpModal(m.actions))` — but NewHelpModal is unexported (newHelpModal)

    Since newHelpModal is unexported, the PoC cannot directly instantiate it. Options:
    1. Export NewHelpModal in tui package — cleanest, reusable
    2. PoC creates its own simplified help view — defeats the purpose of aligning with rootModel

    **Decision: Export NewHelpModal** — add an exported constructor in help.go:
    ```go
    func NewHelpModal(actions *ActionManager) modalView {
        return newHelpModal(actions)
    }
    ```
    This returns modalView interface (unexported) but the PoC can store it as `any` and call methods through a local interface definition.
