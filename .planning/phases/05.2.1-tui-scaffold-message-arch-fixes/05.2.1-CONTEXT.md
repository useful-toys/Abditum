<decisions>
## Implementation Decisions

### Help Modal — Full Dialog Rewrite
- **D-01:** Rewrite `helpModal` in `help.go` to match `tui-specification-novo.md` §Help specification
- **D-02:** Title: "Ajuda — Atalhos e Ações" (Portuguese, not "Keyboard Shortcuts")
- **D-03:** Sizing: max DS dialog width — min(70 columns, 80% of terminal width), height up to 80% of terminal
- **D-04:** Border: `border.default` (#414868 Tokyo Night) with rounded corners — not hardcoded color "62"
- **D-05:** Action bar at bottom border: "Esc Fechar" — not inline "Esc or F1 close this help" text
- **D-06:** Full DS scroll pattern when content exceeds viewport:
  - Scroll indicators: ↑/↓ arrows on right border, ■ thumb for position
  - Navigation: ↑↓ (line), PgUp/PgDn (page), Home/End (top/bottom)
  - Thumb position: `round(scroll_offset / max_scroll × (border_lines - 1))`
  - Arrow prevails over thumb if they coincide
- **D-07:** DS tokens throughout:
  - Title: `text.primary` bold
  - Group labels: `text.secondary` bold
  - Keys: `accent.primary` (not hardcoded "11")
  - Descriptions: `text.primary` (not hardcoded "240")
  - Scroll indicators/thumb: `text.secondary`
- **D-08:** Help modal pushed via `pushModalMsg` — uses rootModel's existing modal stack (current root.go F1 handler already does this correctly)
- **D-09:** Command bar: empty while Help is open (Help registers no internal actions on the bar) — F1 hidden via `HideFromBar` while modal active
- **D-10:** Within each group, actions ordered by Priority (highest first) — matches spec

### Command Bar — Truncation
- **D-11:** `RenderCommandBar()` must truncate actions when they don't fit in `width`
- **D-12:** F1 anchor always stays visible — space for "F1 Ajuda" is reserved before distributing body actions
- **D-12a [BUG FIXED 2026-04-03]:** Guard off-by-2 — leading `"  "` (2 spaces) was not subtracted from the body budget. Guard changed from `-1` (gap only) to `-3` (-2 padding + -1 min gap). Without this, body overflowed the width and the terminal clipped the F1 anchor mid-render.
- **D-13:** Truncation order: drop lowest-priority body actions first (Priority field governs)
- **D-14:** If even the highest-priority action + F1 don't fit → show only F1
- **D-15:** Separator " · " width must be included in fit calculation
- **D-16:** Recalculate on every render (terminal resize changes which actions fit)

### Spinner Frame Order
- **D-17:** Fix `spinnerFrames` array in `messages.go:127` from `["◐", "◓", "◒", "◑"]` to `["◐", "◓", "◑", "◒"]`
- **D-18:** Frame cycle: 0→1→2→3→0 (◐→◓→◑→◒→◐) — matches design system inventory

### PoC Binary
- **D-19:** Update `cmd/poc-mensagens` to use modal pattern (pushModalMsg/popModalMsg) instead of inline `showHelp` bool — aligns with rootModel architecture

### the agent's Discretion
- Exact scroll thumb rendering algorithm edge cases (when content fits exactly, when only 1 line of content)
- Help modal minimum width when terminal is very narrow (< 40 cols)
- Whether to add tests for the new truncation logic in `actions_test.go`

</decisions>

## Existing Code Insights

- `modalView` interface — already implemented by helpModal, modal.go provides the contract
- `pushModalMsg` / `popModalMsg` — root.go handles these; PoC needs equivalent mechanism
- `lipgloss.Width()` — correct width calculation (excludes ANSI codes) — use for truncation math

- Owner-based action registration: `actions.Register(owner, ...Action)`
- Modal stack: `[]modalView` in rootModel, LIFO, last element = topmost
- Token colors: #414868 (border.default), #a9b1d6 (text.primary), #7aa2f7 (accent.primary), #565f89 (text.secondary)
- DS dialog anatomy: rounded border, title in top border, actions in bottom border

<specifics>
## Specific Ideas

- Help modal should follow the same dialog anatomy as other modals: title in top border, action bar in bottom border
- "Ajuda — Atalhos e Ações" title — Portuguese, not English
- Help modal uses `border.default` (neutral/dialog de consulta) — no semantic severity symbol in title
- F1 should be hidden from command bar while Help modal is active (`HideFromBar: true` while modal is on stack)
- The PoC binary doesn't have rootModel's pushModalMsg machinery — needs a simpler modal simulation or a shared modal helper
- Spinner bug is a one-line fix: swap indices 2 and 3 in the `spinnerFrames` array

</specifics>
