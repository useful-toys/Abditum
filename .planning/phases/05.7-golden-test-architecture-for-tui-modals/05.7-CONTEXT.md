# Phase 05.7: Golden Test Architecture for TUI Modals - Context

**Gathered:** 2026-04-06
**Updated:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Implementar arquitetura de testes golden files para validar rendering visual de 4 componentes TUI (layout, espaçamento, cores, fontes), detectando regressões visuais que testes convencionais não capturam.

**Componentes cobertos:**
1. **Linha de mensagem** (`RenderMessageBar` em `messages.go`)
2. **Linha de ações/comandos** (`RenderCommandBar` em `actions.go`)
3. **Modal de ajuda** (`helpModal` em `help.go`)
4. **Modal de decisões** (`DecisionDialog` em `decision.go`)

Golden files validam **apenas o conteúdo do componente retornado por `View()`/função de render**, não o entorno do terminal. O posicionamento na tela é responsabilidade de `rootModel.Render()` via `lipgloss.Place()` e não é testado aqui.

**Fase cobre apenas Camada 1** — golden files visuais (`.txt.golden` + `.json.golden`) em **2 tamanhos de terminal** (30, 60 cols) + testes de `Update()` para componentes interativos.

</domain>

<decisions>
## Implementation Decisions

### Golden File Format
- **Dois arquivos por caso de teste:** `.txt.golden` (layout visual byte-a-byte) + `.json.golden` (transições de estilo)
- **Arquivo `.txt.golden`:** Output ANSI cru — captura espaçamento, bordas, wrapping, caracteres exatos
- **Arquivo `.json.golden`:** Lista de tuplas `[linha, coluna, fg_hex|null, bg_hex|null, style_array]` — captura cores e estilos de forma canônica
- **Regra de Ouro:** Registrar tupla APENAS quando o estado `(fg, bg, style)` muda visualmente — não registrar transições para estado idêntico ao anterior, não registrar linhas que começam em estado padrão sem mudanças
- **Agnóstico a SGR:** Independente de quantos `\x1b[...m` existirem no output, ou sua ordem — registra apenas o resultado visual final. Cores 256-color e truecolor são normalizadas para hex.

### Parser SGR
- Implementar parser SGR mínimo em `internal/tui/testdata/ansiparser.go`
- Extrai transições visuais de output ANSI
- Normaliza cores 256-color e truecolor para hex
- Registra apenas mudanças reais (não redundâncias)
- Reutilizável por todos os 4 componentes

### Estilos Suportados
- `bold`, `italic`, `underline`, `strikethrough`, `faint`, `blink`, `reverse`
- Array de strings no JSON — só inclui estilos ativos
- `Underline` sub-atributos (style, color) registrados como booleano inicialmente; expandir para objetos se necessário

### Estratégias de Teste por Componente

#### A. Modais com Estado (DecisionDialog, HelpDialog)

Componentes com `Update()` e estado interno. Precisam de **instâncias** configuradas com tamanho e dados antes de chamar `View()`.

**DecisionDialog** — fixtures já existem em `decision_test.go` (`pocKey1` a `pocKeyF`). Cada fixture é clonada para 3 tamanhos:

```go
d := pocKey1()       // Destructive 1-action
d.SetSize(30, 24)    // tamanho do teste
out := d.View()      // golden file
```

**HelpDialog** — fixtures precisam ser criados:

```go
func helpFewActions() *helpModal {
    m := newHelpModal(twoActions())
    m.SetSize(80, 24)
    return m
}

func helpManyActions() *helpModal {
    m := newHelpModal(fiveActionsF1())
    m.SetSize(80, 24)
    return m
}
```

#### B. Funções Puras de Renderização (MessageBar, CommandBar)

Sem estado, sem `Update()`, sem `View()`. Chamada direta com dados + largura.

**MessageBar** — `RenderMessageBar(msg *DisplayMessage, width int) string`:

```go
out := RenderMessageBar(&DisplayMessage{Text: "Salvo com sucesso", Kind: MsgSuccess}, 30)
```

**CommandBar** — `RenderCommandBar(actions []Action, width int) string` (função pura, desacoplada de ActionManager):

```go
actions := []Action{
    {Keys: []string{"n"}, Label: "Novo"},
    {Keys: []string{"d"}, Label: "Deletar"},
}
out := RenderCommandBar(actions, 30)
```

**Refatoração necessária:** `RenderCommandBar` atualmente é método de `ActionManager` — extrair para função pura que recebe `[]Action` + `width`. O call site em `root.go` muda de `m.actions.RenderCommandBar(m.width)` para `RenderCommandBar(m.actions.Visible(), m.width)`.

### Matriz de Cobertura — Linha de Mensagem (`messages`)
- **6 kinds × 2 tamanhos (30, 60 cols) = 12 pares**
- Mensagem única de ~45 chars por kind, texto condizente com cada severidade:
  - **Success:** `"Cofre salvo com sucesso — 12 segredos sincronizados"` (48 chars) — valida `✓` + cor verde
  - **Error:** `"Falha ao salvar o cofre — verifique permissões do arquivo"` (57 chars) — valida `✕` + cor vermelha
  - **Warn:** `"Cofre modificado externamente — revisar antes de salvar"` (56 chars) — valida `⚠` + cor amarela
  - **Info:** `"Cofre aberto — 12 segredos, 3 pastas, 2 modelos"` (48 chars) — valida `ℹ` + cor azul
  - **Busy:** `"Carregando cofre, por favor aguarde..."` (38 chars) — valida `◐` (frame 0) + cor ciano
  - **Hint:** `"Pressione F1 para ver todos os atalhos disponíveis"` (51 chars) — valida `?` + cor neutra
- Em 30 cols todas as mensagens truncam com `…` — golden file captura onde o truncamento ocorre
- Em 60 cols a maioria cabe inteira, Busy e Info cabem, Error/Warn podem truncar levemente
- **Total messages:** 12 pares = 24 arquivos

### Matriz de Cobertura — Linha de Ações (`commandbar`)
- **5 cenários × 2 tamanhos (30, 60 cols) = 10 pares**
- `RenderCommandBar` será função pura — `RenderCommandBar(actions []Action, width int) string`
- Cenários explícitos:
  1. **no-actions** — slice vazio → output vazio
  2. **f1-only** — só F1 anchor → `"  f1 Ajuda"`
  3. **typical** — F2 Novo(80), F3 Abrir(70), F4 Salvar(90), F5 Deletar(50), F1 Ajuda(0) → em 60 cabe tudo, em 30 trunca
  4. **unsorted** — mesmas ações que typical mas passadas em ordem inversa de priority → render deve reordenar por priority desc
  5. **many** — 7 ações + F1 (F2-F8) → trunca mesmo em 60, preserva F1
- Em 30 cols com muitas ações, truncamento remove ações de baixa prioridade até F1 caber
- F1 é detectado por `strings.EqualFold(Keys[0], "f1")` — não é flag especial, é qualquer action com tecla "f1"
- **Total commandbar:** 10 pares = 20 arquivos

### Matriz de Cobertura — Modal de Ajuda (`help`)
- **8 cenários × 1 tamanho cada = 8 pares** (16 arquivos)
- `helpModal` será desacoplado de `ActionManager` — recebe `[]Action` + `groupLabel func(int) string`

**Conjunto 1: 3 actions, tela pequena, sem scroll (2 cenários)**
Modal com altura suficiente para exibir todas as 3 actions sem necessidade de scroll. Valida layout limpo, sem indicadores `↑`/`↓`/`■`.

| # | Golden name | Ações | Width | Height | Scroll | O que captura |
|---|-------------|-------|-------|--------|--------|--------------|
| 1 | `help-3actions-30` | 3 (F2 Novo, F3 Abrir, F4 Salvar) | 30 | 12 | 0 | Layout compacto, truncamento de labels, sem scroll indicators |
| 2 | `help-3actions-60` | 3 (F2 Novo, F3 Abrir, F4 Salvar) | 60 | 12 | 0 | Layout completo, descriptions visíveis, sem scroll indicators |

**Conjunto 2: 15 actions, tela limitada a 10 linhas, com scroll (6 cenários)**
15 actions distribuídas em 3 grupos (ex: "Arquivo", "Edição", "Geral"). Dialog height fixo em 10 linhas (maxUsable capped). Scroll testado em 3 posições para validar regressão recorrente de última linha invisível.

| # | Golden name | Ações | Width | Height | Scroll | O que captura |
|---|-------------|-------|-------|--------|--------|--------------|
| 3 | `help-15actions-top-30` | 15 em 3 grupos | 30 | 16 | 0 (0%) | Topo: indicador `↓` visível, primeira linha do grupo 1 |
| 4 | `help-15actions-top-60` | 15 em 3 grupos | 60 | 16 | 0 (0%) | Topo: indicador `↓` visível, descriptions completas |
| 5 | `help-15actions-mid-30` | 15 em 3 grupos | 30 | 16 | ~40% | Meio: indicador `■` (thumb) visível, grupo do meio |
| 6 | `help-15actions-mid-60` | 15 em 3 grupos | 60 | 16 | ~40% | Meio: indicador `■` visível, descriptions completas |
| 7 | `help-15actions-bottom-30` | 15 em 3 grupos | 30 | 16 | 100% | **Fundo: indicador `↑` visível, última linha deve aparecer** |
| 8 | `help-15actions-bottom-60` | 15 em 3 grupos | 60 | 16 | 100% | **Fundo: indicador `↑` visível, última linha deve aparecer** |

**Fixtures de ações para helpModal:**

```go
// 3 actions — grupo 0 (sem header)
func help3actions() []Action {
    return []Action{
        {Keys: []string{"f2"}, Description: "Novo cofre", Group: 0},
        {Keys: []string{"f3"}, Description: "Abrir cofre", Group: 0},
        {Keys: []string{"f4"}, Description: "Salvar cofre", Group: 0},
    }
}

// 15 actions em 3 grupos
func help15actions() []Action {
    return []Action{
        // Grupo 1: "Arquivo"
        {Keys: []string{"f2"}, Description: "Novo cofre", Group: 1},
        {Keys: []string{"f3"}, Description: "Abrir cofre existente", Group: 1},
        {Keys: []string{"f4"}, Description: "Salvar cofre atual", Group: 1},
        {Keys: []string{"f5"}, Description: "Fechar cofre", Group: 1},
        {Keys: []string{"f6"}, Description: "Exportar cofre", Group: 1},
        // Grupo 2: "Edição"
        {Keys: []string{"ctrl+n"}, Description: "Novo segredo", Group: 2},
        {Keys: []string{"ctrl+d"}, Description: "Duplicar segredo", Group: 2},
        {Keys: []string{"ctrl+x"}, Description: "Cortar segredo", Group: 2},
        {Keys: []string{"ctrl+v"}, Description: "Colar segredo", Group: 2},
        {Keys: []string{"del"}, Description: "Excluir segredo", Group: 2},
        // Grupo 3: "Geral"
        {Keys: []string{"ctrl+c"}, Description: "Copiar valor", Group: 3},
        {Keys: []string{"ctrl+s"}, Description: "Salvar alterações", Group: 3},
        {Keys: []string{"ctrl+f"}, Description: "Buscar segredos", Group: 3},
        {Keys: []string{"ctrl+r"}, Description: "Recarregar cofre", Group: 3},
        {Keys: []string{"f1"}, Description: "Esta ajuda", Group: 3},
    }
}

func groupLabel(grp int) string {
    labels := map[int]string{1: "Arquivo", 2: "Edição", 3: "Geral"}
```

**Cálculo do scroll para 40%:**
- 15 actions → ~18 content lines (3 group headers + 15 action lines)
- usableH = height - 4 (borders) → capped at 10
- maxScroll = totalLines - usableH = 18 - 10 = 8
- scroll40% = round(8 × 0.40) = 3

**Total help:** 8 pares = 16 arquivos

### Matriz de Cobertura — Modal de Decisões (`decision`)
- **10 cenários combinados = 10 pares** (20 arquivos)
- **Estratégia:** NÃO é combinação cartesiana (5 severidades × 3 layouts × 2 tamanhos = 30 pares). Cada cenário combina múltiplas dimensões num único teste, cobrindo todas as variações com o mínimo de arquivos.
- **Dimensões:** 5 severidades, 3 contagens de ações (1/2/3), títulos curtos/longos, body 1/2 linhas, larguras 30/60.

| # | Golden name | Severidade | Ações | Título | Body | Width | O que captura |
|---|-------------|-----------|-------|--------|------|-------|--------------|
| 1 | `decision-destructive-1action-short-30` | Destructive | 1 (Excluir) | Curto: "Excluir segredo" | 1 linha: "Gmail será excluído permanentemente." | 30 | Cor + 1 action + truncamento title |
| 2 | `decision-destructive-2action-long-60` | Destructive | 2 (Excluir/Cancelar) | Longo: "Excluir permanentemente este segredo do cofre atual?" | 2 linhas: "Esta ação não pode ser desfeita. Todos os dados associados serão removidos." | 60 | Cor + 2 actions + wrapping body |
| 3 | `decision-error-3action-short-30` | Error | 3 (Recuperar/Abrir backup/Cancelar) | Curto: "Cofre corrompido" | 1 linha: "O arquivo está corrompido. Deseja tentar recuperar?" | 30 | Cor + 3 actions em espaço apertado |
| 4 | `decision-error-1action-long-60` | Error | 1 (OK) | Longo: "Erro crítico ao acessar o cofre — arquivo danificado" | 2 linhas: "Não foi possível decodificar o arquivo. Verifique se o disco está íntegro e tente novamente." | 60 | Cor + title longo + wrapping body |
| 5 | `decision-alert-2action-short-30` | Alert | 2 (Sim/Não) | Curto: "Sobrescrever?" | 2 linhas: "Já existe um segredo com este nome. Deseja substituir o existente?" | 30 | Cor + wrapping + truncamento |
| 6 | `decision-alert-3action-long-60` | Alert | 3 (Substituir/Renomear/Cancelar) | Longo: "Conflito de nome ao salvar novo segredo no cofre" | 1 linha: "Um segredo chamado 'github-token' já existe." | 60 | Cor + title longo + 3 actions |
| 7 | `decision-informative-1action-short-60` | Informative | 1 (Entendi) | Curto: "Dica" | 1 linha: "Pressione Ctrl+N para criar um novo cofre." | 60 | Minimal case, wide |
| 8 | `decision-informative-2action-long-30` | Informative | 2 (Copiar/Fechar) | Longo: "Segredo copiado para a área de transferência com sucesso" | 2 linhas: "O conteúdo será limpo automaticamente em 30 segundos por segurança." | 30 | Wrapping apertado + 2 actions |
| 9 | `decision-neutral-3action-short-30` | Neutral | 3 (Sim/Não/Talvez) | Curto: "Continuar?" | 1 linha: "Tem certeza que deseja prosseguir?" | 30 | 3 actions no truncamento |
| 10 | `decision-neutral-2action-long-60` | Neutral | 2 (Confirmar/Cancelar) | Longo: "Confirmar alterações pendentes antes de fechar o cofre?" | 2 linhas: "Existem 3 modificações não salvas. Se fechar sem salvar, as alterações serão perdidas." | 60 | Tudo cabe, layout completo |

**Cobertura por dimensão (cada dimensão aparece em ≥2 cenários):**
- Severidades: Destructive(2), Error(2), Alert(2), Informative(2), Neutral(2) ✓
- Ações: 1(3), 2(4), 3(3) ✓
- Título: Curto(5), Longo(5) ✓
- Body: 1 linha(5), 2 linhas(5) ✓
- Largura: 30 cols(5), 60 cols(5) ✓

**Total decision:** 10 pares = 20 arquivos

### Total Geral
- **Componentes:** 4
- **Pares de golden files:** 36 (72 arquivos)
- **Testes de Update:** 16 cenários (8 DecisionDialog + 8 HelpDialog)
- **Refatoração:** `RenderCommandBar` de método → função pura

### Test Runner
- Flag `-update` para regenerar golden files: `go test ./internal/tui/... -update`
- Compara `.txt.golden` byte-a-byte
- Compara `.json.golden` tupla-a-tupla
- Falha clara com diff quando há mismatch

### Naming Convention
- Padrão: `{component}-{variant}-{size}.{txt|json}.golden`
- Exemplos: `messages-success-30.txt.golden`, `commandbar-5actions-f1-60.txt.golden`, `decision-destructive-1action-80.txt.golden`
- Diretório: `internal/tui/testdata/golden/`

### Refatorações Necessárias

**1. `RenderCommandBar` de método → função pura**
- Atual: `func (a *ActionManager) RenderCommandBar(width int) string`
- Novo: `func RenderCommandBar(actions []Action, width int) string`
- Call site em `root.go`: `RenderCommandBar(m.actions.Visible(), m.width)`
- Benefício: testes mais simples, API consistente com `RenderMessageBar`

**2. `helpModal` desacoplado de `ActionManager`**
- Atual: `func newHelpModal(actions *ActionManager) *helpModal` — usa `m.actions.All()` e `m.actions.GroupLabel(grp)`
- Novo: `func newHelpModal(actions []Action, groupLabel func(int) string) *helpModal`
- Métodos afetados: `View()`, `totalLines()`, `buildContentLines()`
- Call site em `root.go`: `newHelpModal(m.actions.All(), m.actions.GroupLabel)`
- Benefício: testes golden não precisam instanciar `ActionManager`; fixtures podem passar `[]Action` diretamente

## Specific Ideas

- "O `.json.golden` deve ser canônico — registra apenas transições visuais, não redundâncias de SGR"
- "VSCode mostra color swatches para hex colors no JSON — legibilidade visual direta"
- "Bug real que motivou esta fase: espaçamento faltando em `renderActionBar()` passou em testes existentes"
- "Golden files validam apenas o conteúdo do componente, não o entorno do terminal"
- "Linha de mensagem: `RenderMessageBar(msg, width)` — função pura, fácil de testar"
- "Linha de ações: `RenderCommandBar(width)` — depende de ActionManager, precisa fixtures com ações variadas"
- "Modal de ajuda: `helpModal.View()` — depende de ActionManager.All(), precisa fixtures com muitos grupos"
- "Modal de decisões: `DecisionDialog.View()` — já tem 15 fixtures pocKey1-pocKeyF"
- "3 tamanhos (30, 60, 80): 30 captura wrapping agressivo/truncamento, 60 intermediário, 80 layout ideal"

</specifics>

## Existing Code Insights

- **Table-driven tests** em `decision_test.go`: padrão `MatrixViewRenders` com sub-tests — golden test runner seguirá mesmo padrão

- **Package structure**: `internal/tui/` contém todos os componentes TUI

### Integration Points
- **Golden test runner** será adicionado aos arquivos de teste existentes (`messages_test.go`, `actions_test.go`, `decision_test.go`, e futuro `help_test.go`)
