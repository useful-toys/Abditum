# Phase 5.1: 05-tui-scaffold-root-model-fix - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Esta fase reescreve completamente o pacote `internal/tui` para alinhar todas as interfaces, serviços compartilhados e lógica de dispatch com o documento canônico `tui-elm-architecture.md`.

A Fase 5 entregou a estrutura de arquivos correta e a segurança nil-pointer no `rootModel`. Porém os contratos de interface ficaram desalignados com a arquitetura final: `childModel` tem 5 métodos em vez de 3, `ActionManager` é um registry passivo sem dispatch, `FlowRegistry`/`FlowContext`/`flowDescriptor` implementam um mecanismo de resolução de flows que a arquitetura alvo abandonou, e o lifecycle de flows (iniciar/terminar) não existe.

Esta fase corrige esses desvios **antes** que a Fase 6 construa telas reais sobre eles.

**O que esta fase implementa:**
- `childModel` simplificado para 3 métodos (remove `Context()` e `ChildFlows()`)
- Nova interface `modalView` separada de `childModel`, com `Shortcuts() []Shortcut`
- Nova interface marcadora `modalResult` para dados sensíveis (senha, caminho)
- `flowHandler` ganha `Init() tea.Cmd`
- `Action` redesenhada: `Keys []string`, `Enabled func() bool`, `Handler func() tea.Cmd`, `Scope ActionScope`
- `ActionManager` com API de owners: `Register(owner, ...Action)`, `ClearOwned(owner)`, `SetActiveOwner(owner)`, `Dispatch(key, inFlowOrModal) tea.Cmd`
- `MessageManager` expandido: `MsgKind` (Info/Warn/Error/Busy/Hint), TTL, `Tick()`, `HandleInput()`, `Show(kind, text, ttl, clearOnInput)`
- `startFlowMsg`/`endFlowMsg`/`endFlow()` — lifecycle de flows completo
- `dialogs.go` expandido com `DialogType`, `DialogOption`, e os tipos que Fase 6 usa: `Message`, `Confirm`, `PasswordEntry`, `PasswordCreate`, `FilePicker`
- `root.go` com dispatch reescrito na ordem correta: `Dispatch` → modais → `activeFlow` (fallback) → child ativo
- `modals []modalView` em vez de `modals []childModel`
- `rootModel` ganha `lastCopyAt`, `lastRevealAt time.Time`
- Todos os 7 stubs de child atualizados para `childModel` de 3 métodos
- `modalModel` e `helpModal` migrados para `modalView`
- `root_test.go` reescrito para testar as novas interfaces
- `FlowRegistry`, `FlowContext`, `flowDescriptor`, `chainFlowMsg` **removidos**
- `preVaultModel` renomeado para `welcomeModel` (alinhamento com `tui-elm-architecture.md`)

**O que esta fase NÃO implementa:**
- Nenhuma tela real (welcome form, password fields, vault tree, secret detail)
- Implementações reais de `PasswordEntry`, `PasswordCreate`, `FilePicker` — apenas assinaturas que retornam `pushModalMsg` com stubs funcionais
- `TextInput`, `Select`, `Ask`, `ConfirmOrCancel` — types não necessários até Fase 7+
- Lógica real de timers (`IsLockExpired`, `IsClipboardExpired`, `IsFieldHideExpired`) — Manager ainda não expõe esses métodos
- Qualquer operação de vault real

</domain>

<decisions>
## Implementation Decisions

### D-01: childModel — 3 métodos apenas

```go
type childModel interface {
    Update(tea.Msg) tea.Cmd
    View() string
    SetSize(w, h int)
}
```

`Context()` e `ChildFlows()` **removidos**. A resolução de flows não depende mais de `FlowContext` — flows são instanciados pela `Action.Handler` closure, que captura o estado do child diretamente. `FlowRegistry`, `FlowContext`, `flowDescriptor` e `chainFlowMsg` são todos removidos.

### D-02: modalView — interface separada

```go
type Shortcut struct {
    Key   string
    Label string
}

type modalView interface {
    Update(tea.Msg) tea.Cmd
    View() string
    Shortcuts() []Shortcut
}
```

Modais **não têm `SetSize()`**. O posicionamento (lipgloss.Place) é responsabilidade exclusiva do `rootModel.View()`. O modal renderiza apenas seu conteúdo (a box), e o compositor o sobrepõe. `Shortcuts()` popula a command bar enquanto o modal está ativo.

`rootModel.modals` passa de `[]childModel` para `[]modalView`.

### D-03: modalResult — interface marcadora para dados sensíveis

```go
type modalResult interface {
    isModalResult()
}
```

Mensagens que carregam dados sensíveis (senha, caminho de arquivo, texto de input) implementam `modalResult`. O `rootModel.Update()` as roteia **exclusivamente** para `activeFlow.Update()` — nunca em broadcast para children ou outros modais.

Tipos concretos (definidos junto com seus dialogs):
```go
type passwordEntryResult  struct { Password []byte; Cancelled bool }
type passwordCreateResult struct { Password []byte; Cancelled bool }
type filePickerResult     struct { Path string;     Cancelled bool }
// textInputResult, selectResult — fases futuras
```

### D-04: flowHandler — ganha Init()

```go
type flowHandler interface {
    Init() tea.Cmd
    Update(tea.Msg) tea.Cmd
}
```

`rootModel` chama `Init()` imediatamente após setar `activeFlow`. Flows com dados pré-preenchidos no construtor pulam etapas em `Init()`. O pattern é: se campo já preenchido → `Init()` emite o modal do próximo passo; se vazio → emite o modal do primeiro passo.

### D-05: Action — redesenhada

```go
type ActionScope int
const (
    ScopeLocal  ActionScope = iota  // só quando !inFlowOrModal
    ScopeGlobal                      // sempre — mesmo durante flow/modal
)

type Action struct {
    Keys        []string
    Label       string
    Description string
    Group       string
    Scope       ActionScope
    Enabled     func() bool
    Handler     func() tea.Cmd
}
```

- `Keys[0]` aparece na command bar; todas as keys disparam a ação
- `Enabled` é closure sobre o child — captura ponteiro, sempre lê estado atual
- `Handler` retorna Cmd de operação simples (mutations.go) ou `startFlowMsg{flow: ...}`
- `HideFromBar`, `Priority`, `Disabled` **removidos**
- `ScopeGlobal` substitui interceptação hardcoded de `ctrl+q` e `?` em `dispatchKey`

### D-06: ActionManager — API completa com owners

```go
type ActionManager struct { ... }

func NewActionManager() *ActionManager
func (a *ActionManager) Register(owner any, actions ...Action)
func (a *ActionManager) ClearOwned(owner any)
func (a *ActionManager) SetActiveOwner(owner any)
func (a *ActionManager) Dispatch(key string, inFlowOrModal bool) tea.Cmd
func (a *ActionManager) Visible() []Action  // Enabled()==true, para command bar
func (a *ActionManager) All() []Action      // todas, para help overlay
func (a *ActionManager) RenderCommandBar(width int) string
```

**`Dispatch(key, inFlowOrModal)`** — avalia cada ação na ordem:
1. Verifica `Scope`: `ScopeGlobal` sempre; `ScopeLocal` só quando `!inFlowOrModal`
2. Verifica `Enabled()`: se `false`, pula
3. Compara key: se `key` está em `Action.Keys`, executa `Handler()` e retorna o Cmd

**Owner priority:** quando dois children estão vivos (ex: `vaultTree` e `secretDetail`), `SetActiveOwner(owner)` faz `Dispatch` priorizar ações do dono ativo. Ações do `rootModel` (registradas sem owner ou com owner=rootModel) são sempre elegíveis.

**Invariante de lifecycle:** `ClearOwned(child)` deve ser chamado **antes** de setar o child para `nil`. As closures de `Enabled`/`Handler` capturam o ponteiro do child — `ClearOwned` garante que as ações são removidas antes do ponteiro ser descartado.

### D-07: MessageManager — expandido com kinds e TTL

```go
type MsgKind int
const (
    MsgInfo  MsgKind = iota
    MsgWarn
    MsgError
    MsgBusy
    MsgHint
)

type DisplayMessage struct {
    Text  string
    Kind  MsgKind
    Frame int  // índice de animação para MsgBusy (0-3)
}

func (mm *MessageManager) Show(kind MsgKind, text string, ttlSeconds int, clearOnInput bool)
func (mm *MessageManager) Clear()
func (mm *MessageManager) Current() *DisplayMessage  // nil = sem mensagem
func (mm *MessageManager) Tick()         // chamado pelo rootModel no tickMsg
func (mm *MessageManager) HandleInput()  // chamado pelo rootModel no KeyPressMsg
```

**Regras:**
- `Show()`/`Clear()` **apenas dentro de `Update()`** — nunca em Cmd factories (race condition)
- Last-write-wins — sem fila, sem stack de mensagens
- `MsgBusy` não tem TTL — permanece até ser substituído (flows devem limpar em todo caminho de saída)
- `ttlSeconds == 0` → permanente (sem expiração automática)
- `clearOnInput == true` → `HandleInput()` limpa a mensagem no próximo keypress

### D-08: Lifecycle de flows — startFlowMsg / endFlowMsg

```go
type startFlowMsg struct{ flow flowHandler }
type endFlowMsg   struct{}

func endFlow() tea.Cmd {
    return func() tea.Msg { return endFlowMsg{} }
}
```

`rootModel.Update()`:
```go
case startFlowMsg:
    m.modals = m.modals[:0]   // limpa modais órfãos
    m.activeFlow = msg.flow
    return m, m.activeFlow.Init()

case endFlowMsg:
    m.activeFlow = nil
```

`FlowRegistry`, `FlowContext`, `flowDescriptor`, `chainFlowMsg` **removidos** — flows são iniciados exclusivamente via `Action.Handler → startFlowMsg`.

### D-09: Ordem de dispatch em rootModel.Update()

```go
case tea.KeyPressMsg:
    m.messages.HandleInput()
    m.lastActionAt = time.Now()
    key := msg.String()
    inFlowOrModal := m.activeFlow != nil || len(m.modals) > 0

    if cmd := m.actions.Dispatch(key, inFlowOrModal); cmd != nil {
        return m, cmd
    }
    if len(m.modals) > 0 {
        return m, m.modals[len(m.modals)-1].Update(msg)
    }
    if m.activeFlow != nil {
        return m, m.activeFlow.Update(msg)
    }
    if child := m.activeChild(); child != nil {
        return m, child.Update(msg)
    }
    return m, nil
```

**Justificativa da ordem modais→flow:** flows operam via modais. Se o flow recebesse input antes do modal, teclas digitadas num modal de senha iriam para o flow (que as ignora) e o modal nunca receberia input.

### D-10: dialogs.go — tipos necessários para Fase 6

`DialogType` e `DialogOption`:
```go
type DialogType int
const (
    DialogQuestion DialogType = iota
    DialogAlert
    DialogInfo
)

type DialogOption struct {
    Label   string
    Keys    []string
    Cmd     tea.Cmd
    Default bool  // ENTER aciona esta opção
    Cancel  bool  // ESC aciona esta opção
}
```

Factories implementadas:
```go
func Message(dtype DialogType, title, message string) tea.Cmd
func Confirm(dtype DialogType, title, message string, onYes, onNo tea.Cmd) tea.Cmd
func PasswordEntry(title string) tea.Cmd    // → passwordEntryResult
func PasswordCreate(title string) tea.Cmd  // → passwordCreateResult
func FilePicker(title string, mode FilePickerMode, extension string) tea.Cmd  // → filePickerResult
```

`Ask` e `ConfirmOrCancel` — **adiados** para a fase que introduz seu primeiro caso de uso.

`passwordEntryModal`, `passwordCreateModal`, `filePickerModal` são stubs que implementam `modalView` — `Update()` retorna `popModalMsg{}` imediatamente (placeholder). Implementação real na Fase 6.

<specifics>
## Specific Ideas

- O documento `tui-elm-architecture.md` contém pseudocódigo completo para `rootModel.Update()`, `broadcast()`, `tickMsg` handler, e exemplos de flow completo (`openVaultFlow`). O implementador deve seguir esses exemplos diretamente.
- `modalView` sem `SetSize` implica que o `rootModel.View()` faz o overlay: `lipgloss.Place(m.width, m.height, Center, Center, modal.View())` — o modal não sabe onde está na tela.
- `Action.Enabled` é closure sobre ponteiro do child — Go garante que closure captura o ponteiro, então `m.focused` é sempre o valor atual na chamada, não no registro.

</specifics>
