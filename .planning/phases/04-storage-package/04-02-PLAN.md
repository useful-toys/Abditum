    **`SaveNew(destPath string, cofre *vault.Cofre, password []byte) error`** (ATOMIC-03):
    - Call `vault.SerializarCofre(cofre)` → JSON bytes
    - Call `crypto.GenerateSalt()` → 32-byte salt
    - Look up profile: `ProfileForVersion(CurrentFormatVersion)` → params
    - Call `crypto.DeriveKey(password, salt, params.ToArgonParams())` → key

    - Actually: build 49-byte header buffer. Write Magic to [0:4], CurrentFormatVersion to [4], salt to [5:37]. Then call `crypto.EncryptWithAAD(key, jsonBytes, headerWithPlaceholderNonce)` — wait, AAD needs the full header including nonce. Correct approach:
      1. Generate salt
      2. Derive key
      3. Call `crypto.EncryptWithAAD(key, jsonBytes, nil)` first to get nonce — NO, AAD MUST be the full header
      4. **Correct approach**: Build header with zeros for nonce, call EncryptWithAAD with header-with-zero-nonce as AAD — NO, this would mean load needs to use zero-nonce header as AAD too, which defeats purpose
      5. **ACTUAL correct approach per spec**: The nonce IS in the header. So: generate salt, generate nonce separately (`io.ReadFull(rand.Reader, nonce)`), build complete 49-byte header, use header as AAD, then encrypt with `gcm.Seal` directly. BUT we designed EncryptWithAAD to generate nonce internally...

    **Resolution**: EncryptWithAAD generates its own nonce. The full header (including that nonce) is the AAD. This creates a chicken-and-egg: we need the nonce to build the AAD, but AAD is passed to encrypt which generates the nonce. TWO OPTIONS:
    - **Option A**: EncryptWithAAD returns nonce, we build header with that nonce, but the AAD during encryption didn't include the real nonce — BROKEN
    - **Option B**: Add `EncryptWithAADAndNonce(key, plaintext, nonce, aad []byte)` that accepts a pre-generated nonce — but this violates crypto best practice of generating nonce internally
    - **Option C**: Storage generates nonce itself, builds full header as AAD, then calls a lower-level encrypt. BUT we want to keep crypto responsibility in crypto package.
    - **Option D (CORRECT)**: The AAD authenticates everything EXCEPT what GCM already authenticates. GCM already binds the nonce to the ciphertext (nonce is input to GCM). So the AAD should be `magic + version + salt` (37 bytes), NOT the full 49-byte header. The nonce is authenticated by GCM's own mechanism. **Wait — the spec says "O cabeçalho completo (magic + versão_formato + salt + nonce) é autenticado como AAD"** — this means the full 49 bytes.

    **FINAL approach**: Make the crypto API flexible enough. Since EncryptWithAAD generates nonce internally and returns it, storage can:
    1. Generate salt via `crypto.GenerateSalt()`
    2. Derive key
    3. Call `crypto.EncryptWithAAD(key, jsonBytes, aadPlaceholder)` where aadPlaceholder uses a temp nonce — NO THIS IS WRONG
    
    **SIMPLEST CORRECT approach**: Storage generates its own nonce, builds the full 49-byte header, uses it as AAD, then calls crypto at a lower level. To avoid bypassing crypto package:
    - Add `EncryptWithNonceAndAAD(key, plaintext, nonce, aad []byte) ([]byte, error)` to crypto — accepts externally-provided nonce
    - Storage generates nonce via `crypto/rand`, builds full header, calls this function
    
    **ALTERNATIVELY**: Modify `EncryptWithAAD` to accept an optional nonce parameter, or have two variants. The cleanest: keep `EncryptWithAAD` as-is (generates nonce, returns it) and in storage reconstruct the AAD post-hoc for verification purposes — NO, AAD must be known at encrypt time.

    **DECISION**: Refine the crypto API. Instead of `EncryptWithAAD(key, plaintext, aad) → (nonce, ciphertext, error)`, implement as:
    - `EncryptWithAAD(key, plaintext, nonce, aad []byte) ([]byte, error)` — caller provides nonce AND aad
    - This matches `DecryptWithAAD(key, ciphertext, nonce, aad)` signature symmetrically
    - Storage generates nonce via `io.ReadFull(crypto/rand.Reader, nonce)`, builds header with that nonce, passes both to EncryptWithAAD

    **With corrected API, SaveNew flow:**
    1. `vault.SerializarCofre(cofre)` → jsonBytes
    2. `crypto.GenerateSalt()` → salt
    3. Generate nonce: `nonce := make([]byte, NonceSize); io.ReadFull(rand.Reader, nonce)`
    4. Build 49-byte header: Magic[0:4] + CurrentFormatVersion[4] + salt[5:37] + nonce[37:49]
    5. `ProfileForVersion(CurrentFormatVersion)` → profile → `profile.ToArgonParams()` → params
    6. `crypto.DeriveKey(password, salt, params)` → key; `defer crypto.Wipe(key)`
    7. `crypto.EncryptWithAAD(key, jsonBytes, nonce, header)` → ciphertext (includes GCM tag)
    8. Write `header + ciphertext` to destPath using `os.WriteFile` with `0600` permissions
    9. Wipe jsonBytes: `crypto.Wipe(jsonBytes)`

    **`Save(vaultPath string, cofre *vault.Cofre, password, salt []byte) error`** (ATOMIC-01, ATOMIC-02):
    - Note: Save reuses the existing salt (salt only changes on password change). Password and salt are passed by the caller (Manager holds them in session).
    1. `vault.SerializarCofre(cofre)` → jsonBytes
    2. Generate fresh nonce: `nonce := make([]byte, NonceSize); io.ReadFull(rand.Reader, nonce)`
    3. Build 49-byte header with existing salt + new nonce
    4. Derive key, encrypt with AAD (same as SaveNew steps 5-7)
    5. Atomic write protocol:
       a. `tmpPath := vaultPath + ".tmp"`
       b. Write `header + ciphertext` to tmpPath with `0600` permissions
       c. Backup rotation: if `.bak` exists → rename `.bak` to `.bak2`; rename `vaultPath` to `.bak`
       d. `atomicRename(tmpPath, vaultPath)` — platform-specific
       e. If rename succeeds → delete `.bak2` if exists (best-effort, ignore error)
       f. If any step after tmpPath write fails → `os.Remove(tmpPath)` (cleanup)
    6. Wipe key and jsonBytes

    **`Load(vaultPath string, password []byte) (*vault.Cofre, FileMetadata, error)`**:
    1. Read entire file: `data, err := os.ReadFile(vaultPath)`
    2. Compute FileMetadata: `meta.Size = int64(len(data))`, `meta.Hash = sha256.Sum256(data)`
    3. Validate length ≥ HeaderSize (49) → `ErrInvalidMagic` if too short
    4. Check magic: `data[0:4]` must equal `Magic` → `ErrInvalidMagic`
    5. Read version: `data[4]` → if > `CurrentFormatVersion` → `ErrVersionTooNew`
    6. Read salt: `data[SaltOffset:SaltOffset+SaltSize]`
    7. Read nonce: `data[NonceOffset:NonceOffset+NonceSize]`
    8. Extract header: `data[0:HeaderSize]` (for AAD)
    9. Extract ciphertext: `data[HeaderSize:]`
    10. `ProfileForVersion(version)` → params
    11. `crypto.DeriveKey(password, salt, params.ToArgonParams())` → key; `defer crypto.Wipe(key)`
    12. `crypto.DecryptWithAAD(key, ciphertext, nonce, header)` → jsonBytes; on error → `crypto.ErrAuthFailed` (wrong password)
    13. `vault.DeserializarCofre(jsonBytes)` → cofre; on error → `ErrCorrupted`
    14. Wipe key, wipe jsonBytes
    15. Return `(cofre, meta, nil)`

    Use fast Argon2id params in tests (Time=1, Memory=64*1024, Threads=1) for speed. Define a `testParams` or `FastParams` helper.

    **`internal/storage/atomic_rename_unix.go`:**
    ```go
    //go:build !windows

    package storage

    import "os"

    // atomicRename performs an atomic file rename on Unix systems.
    // POSIX rename(2) is atomic when source and destination are on the same filesystem.
    // The .tmp file MUST be in the same directory as the target (filepath.Dir) to guarantee
    // same-filesystem operation — never use os.TempDir() which may be a different mount.
    func atomicRename(src, dst string) error {
        return os.Rename(src, dst)
    }
    ```

    **`internal/storage/atomic_rename_windows.go`:**
    ```go
    //go:build windows

    package storage

    import (
        "golang.org/x/sys/windows"
    )

    // atomicRename performs an atomic file rename on Windows.
    // Standard os.Rename on Windows is NOT atomic when replacing an existing file —
    // it can leave the destination deleted if the process is interrupted between
    // the internal DeleteFile and MoveFile calls.
    // MoveFileEx with MOVEFILE_REPLACE_EXISTING provides atomic replace semantics
    // on NTFS (the rename is a single metadata operation in the MFT).
    func atomicRename(src, dst string) error {
        srcPtr, err := windows.UTF16PtrFromString(src)
        if err != nil {
            return err
        }
        dstPtr, err := windows.UTF16PtrFromString(dst)
        if err != nil {
            return err
        }
        return windows.MoveFileEx(srcPtr, dstPtr, windows.MOVEFILE_REPLACE_EXISTING)
    }
    ```

    Both files export the same `atomicRename(src, dst string) error` function — the build system selects the correct one at compile time.
