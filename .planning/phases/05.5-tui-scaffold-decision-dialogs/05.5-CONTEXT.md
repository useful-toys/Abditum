<domain>
## Phase Boundary

Implement and validate the foundational **decision dialog infrastructure** — the core modal type used for confirmations, acknowledgments, and error notifications. This phase scaffolds **zero domain logic** (no vault integration); instead, it focuses on proving the dialog mechanism works correctly: rendering, theming, interaction, and message/action integration.

### What IS in scope:
- DecisionDialog struct (inert data model)
- View() renderer following DS and spec exactly
- Severity × Intention matrix (5 severities, 2 intentions, with test fixtures for all combinations)
- Interaction model: Enter/Esc dispatch, action selection
- MessageBar integration (shown before dialog closes; cleared on close)
- ActionBar integration (shows internal actions, hides external action manager)
- Unit tests covering all 10 combinations + edge cases (long messages, wrapping, button overflow)

### What IS NOT in scope:
- Integration with vault domain (D-25, flows, etc.)
- FilePicker dialog
- PasswordEntry dialog
- Interactive flows (opening cofre, creating cofre)
- Domain-driven icon handling (these are mocked)

<decisions>
## Implementation Decisions

### DecisionDialog Type Hierarchy

```
DecisionDialog
  ├─ Severity: {Destrutivo, Erro, Alerta, Informativo, Neutro}
  ├─ Intention: {Confirmação, Reconhecimento}
  ├─ Title: string
  ├─ Message: string (wrapped by renderer)
  ├─ Actions: []DecisionAction
  ├─ Internal actions: for message bar interaction?
```

**Per spec 'Diálogos de Decisão':**
- Severity (não-ortogonal a Intention; defines border + title token + default key token)
- Intention (Confirmação = multiple choice; Reconhecimento = Enter OK only)
- Symbol per severity (⚠ ✕ ⚠ ℹ — or none for Neutro)
- Message in `text.primary`; referenced item names in **bold**
- Action bar at bottom: default action **bold**, others normal, Esc always last

### Severity × Intention Matrix

Test fixtures for all **10 combinations**:

| Severity | Confirmação | Reconhecimento |
|----------|-------------|----------------|
| **Destrutivo** | ⚠ Delete secret | ⚠ Confirm deletion? |
| **Erro** | ✕ Retry operation | ✕ Operation failed |
| **Alerta** | ⚠ Abandon unsaved? | ⚠ Locked — too many attempts |
| **Informativo** | ℹ Overwrite file? | ℹ Conflict detected |
| **Neutro** | — Continue? | — (no typical ackn use) |

Each combination has:
- Correct border token (severity-driven)
- Correct default action token (severity-driven)
- Correct symbol (severity-driven; Neutro has none)
- Correct action layout (Confirmação = 3 actions; Reconhecimento = 1 action + Esc)

### Rendering Strategy

```
renderFrame(modal)
  ├─ workArea (background; contains modal overlay via lipgloss.Place)
  │   └─ modal.View() → DecisionDialog.renderDialog()
  │       ├─ border lines
  │       ├─ title (symbol + text)
  │       ├─ message (wrapped to dialog width)
  │       └─ action bar (Enter/Esc with token colors)
  ├─ msgBar (visible, showing dip/error/info from DecisionDialog closure)
  └─ cmdBar (shows only F1 Ajuda; action manager hidden)
```

Per spec:
- Dialog centered via lipgloss.Place (inside workArea)
- Message wrapped to 70 cols max (or 80% terminal, whichever smaller)
- Long messages scroll vertically if needed (future iteration)
- Action bar follows DS ordering: default left, cancel right, intermediates between

### Interaction Model

```
DecisionDialog.Update(msg tea.Msg)
  ├─ tea.KeyPressMsg
  │   ├─ "enter" → dispatch default action → popModalMsg + cmd
  │   ├─ "esc" → dispatch cancel action → popModalMsg + cmd
  │   └─ "←"/"→" → navigate between actions (Confirmação only)
  ├─ tickMsg → (no animation; acknowledge)
  └─ other → nil
```

**Action dispatch:**
- Spec says "Ações de confirmação/cancelamento já estão na borda do diálogo — não são duplicadas na barra"
- So DecisionDialog.Shortcuts() returns empty slice (no action bar display)
- Enter/Esc are implicit to dialog type, not registered as actions

### MessageBar Integration

Spec rule: "Diálogos de decisão (confirmação/reconhecimento) **não** usam a barra — a mensagem completa está no corpo do diálogo."

**BUT:** after dialog closes, if an error occurred (e.g., "file not found"), the orquestrador emits a message.

Implementation: DecisionDialog emits **zero** messages. The parent flow/orquestrador handles post-close messaging.

### ActionBar During Dialog

Per spec: "A barra mostra apenas as ações internas do diálogo... Ações de confirmação e cancelamento já estão na borda do diálogo — não são duplicadas na barra."

Implementation: When DecisionDialog is active (top of modal stack), **rootModel.renderFrame** routes to `modal.Shortcuts()`. Since DecisionDialog.Shortcuts() returns empty, cmdBar shows only F1 Ajuda (+ empty space if no F1).

### Test Coverage Strategy

**Unit tests in decision_test.go:**

1. **Type construction**: NewDecisionDialog with title, message, actions
2. **Rendering**: View() produces correct border token, symbol, text layout for all 10 combinations
3. **Interaction**: Update() handles Enter/Esc/← →; emits popModalMsg
4. **Message wrapping**: Long messages wrap at 70 cols, multi-line display
5. **Action layout**: buttons in correct order, correct token colors
6. **Edge cases**:
   - Empty action list (should not occur; guard with panic)
   - Very long button labels (truncation strategy)
   - Single-char title (padding correct)

**Integration test in root_test.go:**

- Modal stack: push DecisionDialog → View() shows overlay → Update() pops on Enter
- Command bar: F1 shows (action manager hidden during dialog)
- Message bar: remains visible; dialog doesn't write to it

### DecisionDialog struct (decision.go)

```go
type Severity int
const (
  SevereDestrutivo Severity = iota
  SevereErro
  SevereAlerta
  SevereInformativo
  SevereNeutro
)

type Intention int
const (
  IntentConfirm Intention = iota
  IntentAcknowledge
)

type DecisionAction struct {
  Key   string // "enter", "esc", "s", "n"
  Label string // "OK", "Cancelar", "Salvar"
  IsDefault bool
  IsCancel  bool
  // Handler will be added in Phase 06 (domain integration)
}

type DecisionDialog struct {
  title       string
  message     string
  severity    Severity
  intention   Intention
  actions     []DecisionAction
  width       int      // from SetSize
  height      int      // from SetSize
}
```

### Rendering Checklist (decision.go View())

- [ ] Border and title use correct severity token
- [ ] Symbol present/absent based on severity
- [ ] Message wrapped to max width
- [ ] Message lines centered or left-aligned per DS
- [ ] Action buttons in correct order (default left, cancel right)
- [ ] Button tokens correct per severity + intention
- [ ] Default action in **bold**; others normal
- [ ] Frame occupies exact modal dimensions (SetSize enforced)

### Interaction Checklist (decision.go Update())

- [ ] Enter key dispatches default action
- [ ] Esc key dispatches cancel action (or OK for Acknowledge)
- [ ] ← → navigate between actions (Confirmação only)
- [ ] All other keys ignored (no bubbling)
- [ ] popModalMsg emitted on action dispatch
- [ ] No other Cmd returned

### Test Checklist (decision_test.go)

1. **Matrix coverage**: All 10 combinations render + interact correctly
2. **Edge cases**: Long title, long message, long button label, empty actions (panic)
3. **Token verification**: Check that View() output contains correct ANSI color codes for tokens
4. **Action dispatch**: Verify Update(KeyPressMsg) routes to correct action
5. **Integration**: Modal stack push/pop with root_test.go helpers
