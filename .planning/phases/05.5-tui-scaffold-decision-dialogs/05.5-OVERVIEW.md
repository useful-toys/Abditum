## What is Phase 05.5?

Implement and validate the **decision dialog** — the core modal type used for user confirmations, acknowledgments, and error notifications throughout the TUI.

This phase focuses on **pure TUI infrastructure**: rendering, styling, keyboard interaction, and integration with the modal stack. **Zero domain logic** — no vault integration, no flows, just the dialog itself.

## Key Deliverables

✅ **DecisionDialog type** with Severity × Intention matrix (5 severities, 2 intentions)  
✅ **View() rendering** following DS + spec exactly  
✅ **Update() interaction** (Enter/Esc/arrows, modal dispatch)  
✅ **All 10 test fixtures** covering the full combinations  
✅ **10+ unit tests** + 1 integration test  
✅ **No changes to existing code** (pure addition)

## Scope Boundary

### IN SCOPE
- DecisionDialog struct (inert data)
- View() renderer (DS/spec compliant)
- Update() for keyboard input
- All 10 severity × intention combinations
- Message bar visibility (but not writing)
- Action bar behavior (F1 only during dialog)

### OUT OF SCOPE (Phase 06+)
- Domain integration (flows, vault operations)
- Handler closures (deferred to Phase 06)
- PasswordEntry dialog (functional dialog type)
- FilePicker dialog (complex multi-panel)
- Animation/transitions
- Screen reader integration

## Why This Phase?

Phase 05 completed the TUI scaffold (root, actions, messages). Phase 05.4 validated action architecture. **Phase 05.5 closes the loop** by adding the dialog infrastructure itself.

Dialogs are prerequisites for Phase 06 (welcome screen, vault open/create flows) — they're the mechanism for user confirmation, error reporting, and user input.
